Premortem — Context-Atoms Design

2026-05-28 · Following the /premortem skill · 5 phases · 8 parallel deep-dives

The design under premortem

An agent-context-persistence system replacing heavy SDD-style specs with atomic, anchored, kind-typed memory. Storage: repo-local markdown files in docs/context/{intent,decisions,constraints}/, one fact per file (30–80 lines), with YAML frontmatter (kind, anchors, tags, links, status). Files live in git, reviewed in PRs.

Retrieval: an interview skill produces a structured envelope {files, subsystems, tags, flows}. A pure-function selector runs anchor∩files + tag∩tags + one-hop graph walk + dedupe + order by kind. Agent has escape-hatch context.search/context.walk tools.

Lifecycle: post-task distill proposes atoms; PR-touched-anchors checklist gates curation; churn-based staleness. Packaging: standalone CLI + MCP, tool-neutral across repos.

Key prior evidence: The ETH Zurich + LogicStar.ai AGENTS.md study (arXiv 2602.11988) found flat repo-level context files reduced task success in 5/8 settings and added 20%+ cost. The design bets that selective, scoped injection behaves fundamentally differently — this is unverified.

Headline findings

Most probable failure
Slow fade: curation collapse + anchor rot

Curation slows after sprint 1; refactors break anchors faster than they're repaired; the PR checkbox becomes ceremonial; the corpus silently rots. Doesn't require any single dramatic failure — only that cumulative curation cost exceeds visible value.

Most damaging failure
Empirical disproof: atoms prime exploration regardless of selectivity

Telemetry shows agents that receive even one atom open 2.3× more files in first 5 tool calls. The AGENTS.md study's finding generalizes — selectivity doesn't save us. If true, the design is a polished version of a known-bad pattern.

Central hidden assumption
Selective ≠ flat at the mechanism level

The whole design rides on one bet: that scoped, kind-ordered injection produces fundamentally different agent behavior than always-on flat injection. If false, no architecture saves it. Ship the measurement before the mechanism.

Deep dives

3.1 — DESIGN-FATAL

Empirical disproof: atoms hurt success even when scoped

Failure narrative

Month 0–2: Ship v1 + 40 curated atoms. Distill generates ~6 atoms/PR. Dev feedback positive: "agents quote my decisions back to me."

Month 3: First ablation eval (n=50): atoms-on 72% vs atoms-off 74%, token cost +18%. Within noise. Team tunes anchors, halves average injection.

Month 4–5: Library grows to ~180 atoms across 3 repos. Re-run ablation (n=200): atoms-on 68% vs atoms-off 75%, +24% cost, p<0.01. Simple tasks regress hardest (−9pp).

Month 6: Telemetry: agents receiving even one constraint atom open 2.3× more files in first 5 tool calls. Atoms aren't being used — they're priming exploration. Precision is 81%, but precision isn't the lever. The act of injection itself widens the agent's search frontier. ETH Zurich result reproduced under scoped injection. Design's core bet falsified.

Falsified assumptions

  • Scoping changes the agent's response to injected context — it doesn't.
  • Precision is the right optimization target — receiving curated context at all triggers broader exploration.
  • Token cost scales with atoms injected — it scales with exploration triggered, ~5× direct cost.
  • Dogfooding catches regressions — devs liked it subjectively while it was hurting agents.

Early warning signs

  • Ablation delta crosses −2pp on any task category by month 3.
  • Files-opened-in-first-5-calls ratio (atoms-on / atoms-off) exceeds 1.5×.
  • Token cost overhead >15% on simple tasks (rename/typo/bugfix).
  • Subjective dev satisfaction rising while measured success flat or declining.

Mitigations

  • Gate launch on ablation eval, not lint. Require +3pp success delta before any user sees injected atoms.
  • Default OFF for simple task classes. Inject only for novel/architectural work.
  • Inject by reference, not content — surface atoms as context.search results the agent must fetch.
  • Ship telemetry before content — exploration-ratio metric in week 1, atoms behind a flag.
3.2 — HIGH PROBABILITY

Curation collapse: corpus growth stalls, atoms drift from code

Failure narrative

Week 1: one engineer seeds 22 atoms. Week 2–3: teammates add 8 more. The selector feels magical. Week 4: the first "review affected atoms" checkbox; the author skims, ticks it. Week 6: a refactor renames processPod to runPodLoop; the anchor glob still matches the file so the atom isn't flagged — it now describes a function that doesn't exist. Week 10: the seeder goes on parental leave. Week 12: "is this atom still right?" — nobody knows; PR ships anyway.

Month 4: an agent confidently cites a superseded SQLite-schema atom and produces a wrong migration. Team blames the AI. Month 5: distill still proposes atoms; PRs reject them as "we'll do it later." Month 6: docs/context/ is legacy documentation. The selector still returns results — they're just wrong.

Falsified assumptions

  • Engineers will edit markdown at PR time as a new habit — they won't, it competes with shipping.
  • Lint-passing correlates with accurate — staleness is invisible to syntax.
  • Churn-based staleness catches drift — too noisy, gets ignored.
  • Distill proposals become merges — proposing is cheap; review is the bottleneck.

Early warning signs

  • Atom creation rate drops >60% sprint-over-sprint after sprint 1.
  • "Review affected atoms" checkbox check-time median <3 seconds.
  • Distill-proposed atoms <20% merge rate after 2 weeks.
  • Gap rate climbing while corpus size flat — agents routing around stale atoms.

Mitigations

  • AST-symbol staleness (not just churn) via tree-sitter; surfaces in PR comments, blocks merge until acknowledged.
  • Shift curation into the agent — agent proposes atom edits in same PR as code edits.
  • Atoms decay to archived automatically if unverified for N weeks; force re-verification.
  • Measure outcome, not output — dashboard gap-rate and citation-accuracy, not atom count.
3.3 — MEDIUM

Wiki-ification: atoms become specs in another directory

Failure narrative

Month 1: tight atoms, 20–40 lines. Month 2: first compound atom — decision plus three constraints "because they're conceptually inseparable." Reviewer approves; precedent set. Month 3: atoms grow H2 sections (## Context, ## Decision, ## Tradeoffs); median length climbs from 32 → 51 lines. Month 4: authors notice fat atoms retrieve better than thin ones — the incentive flips. Month 5: new contributors use existing atoms as templates and write at 78 lines. Month 6: selector returns 5 atoms totalling 400 lines of dense mixed prose. Performance degrades exactly like the AGENTS.md monolith the system was built to replace.

Falsified assumptions

  • Authoring discipline is durable under social pressure — reviewers won't reject "a bit long but well-written" from a senior contributor.
  • File-size cap is a proxy for content shape — 80 lines of one idea ≠ 80 lines of four ideas.
  • Atomicity is self-reinforcing — fat atoms outcompete thin ones in retrieval, creating a gradient toward bloat.

Early warning signs

  • Median atom length crosses 50 lines (vs. target 30–40).
  • Heading count per atom >1 — multi-fact tell.
  • Mean atoms-per-envelope drops while mean tokens-per-envelope rises.

Mitigations

  • Lint on shape: reject >1 H2, >N top-level bullets, >1 kind. Cap ~40 lines, not 80.
  • Single-claim check: require claim: frontmatter field; CI LLM-checks body supports exactly that claim.
  • Retrieval anti-incentive: selector penalizes atom length, removing the gradient toward bloat.
  • Weekly corpus audit with p50/p90 length and "drift score."
3.4 — LOW PROBABILITY · MEDIUM DAMAGE

Kind collapse: intent / decision / constraint blur

Failure narrative

Month 1: clean. intent says "Pod tokens exist because…", decision says "We chose HMAC over JWT because…", constraint says "Pod tokens must be <2KB…" Month 3: intent/network-isolation.md contains intent + decision fused; reviewer splits it; the splits feel redundant. Month 6: constraint/migration-numbering.md opens with three paragraphs of why before stating the rule. decision/sqlite-over-postgres.md ends with a constraint. New contributors copy whichever neighbor file they opened first. Kind becomes a coin flip. PR reviews argue about kind assignment for atoms whose content everyone agrees on. The selector orders intent → decision → constraint, but the actual content order is scrambled. Nobody notices because outputs still look plausible.

Falsified assumptions

  • "Why exists" / "why chose" / "what must hold" are cleanly separable — every decision has embedded intent, every constraint a rationale.
  • Authors reach for the right kind without a forcing function — they don't.
  • Review catches mis-kinding — reviewers themselves disagree.
  • Selector ordering is robust to ~10% mis-kinding — even 30% silently corrupts the narrative.

Early warning signs

  • Kind-change rate in PR review >15%.
  • Inter-rater agreement test (5 engineers, 10 kind-stripped atoms): <70% means folk-taxonomy.
  • Keyword leak: "because" in constraint/, "must" in intent/.

Mitigations

  • Collapse to two kinds: rationale (why, including chosen-because) and constraint (invariants). Intent-vs-decision is blurry; constraint is sharp.
  • Structural templates per kind — mandatory sections enforced by lint.
  • Promote ordering out of kind: add depends_on: field; selector follows declared dependencies, not taxonomy.
  • Inter-rater calibration in CI: periodic LLM re-classification of kind-stripped atoms.
3.5 — HIGH PROBABILITY

Anchor rot: refactors break anchors silently

Failure narrative

Month 1: clean. Month 2: someone splits pod-manager.ts into two files; the atom's anchor breaks; the author has no context, adds a lint-disable to unblock CI. Three sprints later a daemon directory reorg moves 40 files; 60 atoms break at once; nobody owns the backlog. Someone "fixes" it by rewriting specific anchors to packages/daemon/src/pods/** — silences the lint but now every pod touching pods/ surfaces 6 atoms, 4 irrelevant. Reviewers see the checklist fire on a typo fix, dismiss it, muscle memory sticks. By month 6 the checklist is decorative.

Falsified assumptions

  • Refactor authors update anchors as part of the refactor — they don't.
  • Lint warnings on orphans create enough pressure to fix them — they accumulate; one PR blocked by an unrelated orphan teaches everyone to disable.
  • Glob anchors are a reasonable fallback — they collapse signal-to-noise of the checklist, which was the value.
  • Anchor correctness degrades gradually — it doesn't, directory reorgs invalidate dozens at once.

Early warning signs

  • Orphan rate >5% of total atoms, or growing >2pp/month.
  • Lint-disable comments referencing anchor rules in >1 PR/week.
  • Ratio of git mv to anchor-updating commits drifts above 10:1.
  • Glob-anchor share climbs past 30%.

Mitigations

  • Symbol-based anchors via tree-sitter (pods/pod-manager.ts#processPod) survive splits and moves.
  • Git hook detects renames via git log --follow -M and auto-rewrites anchors.
  • Anchor freshness signal (last-verified commit SHA); selector deprioritizes stale.
  • Block lint-disable: orphans become CI errors after 7-day grace.
3.6 — MEDIUM

Envelope weakness: fuzzy→structured conversion is unreliable

Failure narrative

Week 1–4: envelopes look fine; humans are paying attention. Week 5–12: devs notice envelope confirmation is the slowest step of /prep and start mashing "looks good." The LLM extractor invents flows that don't exist as canonical tags; the validator only checks shape, not membership. subsystems get plausible-but-wrong labels (orchestration vs canonical pod-lifecycle). The taxonomy itself drifts. Week 13–26: selector retrieves atoms tagged orchestration for a pod-lifecycle query and misses everything relevant. Agents learn the injected context is noise and start every task with grep. The push channel becomes decorative.

Falsified assumptions

  • A canonical taxonomy stays stable enough that envelopes authored months apart remain comparable.
  • LLM extraction + schema validator catches semantic errors — schema only validates shape, not correctness.
  • Humans meaningfully review LLM-proposed envelopes during interview.
  • flows is a learnable category — it's a fuzzy human concept with no ground truth.

Early warning signs

  • Envelope-field correction rate <15% after week 4 (rubber-stamping).
  • LLM-extractor F1: subsystems <0.75, flows <0.5.
  • Tag cardinality growing faster than tag reuse.

Mitigations

  • Closed-vocabulary tags/subsystems at validation time; new value requires explicit taxonomy-update PR.
  • Drop flows or demote to free-text the selector ignores.
  • Weight retrieval primarily on files + path-derived subsystems (both mechanical); LLM fields are tiebreakers.
  • Continuous per-field F1 monitoring; auto-disable fields whose F1 falls below threshold.
3.7 — MEDIUM

Selector calibration: one-hop is wrong-sized

Failure narrative

Month 1: one-hop works on the demo. Month 2: a refactor decision supersedes an earlier decision that itself refines a principle — the principle is two hops away and gets silently violated. Fix: "walk supersedes chains transitively." A special case. Month 3: contradicts links fire across the graph; weight them lower. Now there's a weights table. Month 4: per-kind hop budgets appear. The selector grows a config file. Month 5: agents call pull on every task as a safety net — push-primary is dead. Month 6: the "deterministic, debuggable selector" requires reading 400 lines of weighting logic plus a YAML. Three engineers have opinions about the weights. Nobody wants to own it.

Falsified assumptions

  • "One hop" is a property of the graph — actually a property of each link-type.
  • Parent intent lives one hop away — chains in real codebases are 3–5 deep.
  • Link semantics stay clean — authors conflate related/implements and contradicts/supersedes.
  • Determinism implies debuggability — determinism without locality buys only reproducibility.

Early warning signs

  • pull_context invocation rate >30% of pods.
  • Avg atoms-injected-per-pod drifting upward (8 → 18).
  • Selector config gains commits at >1/week.
  • "Why was this included?" questions in incidents take >5 min to answer.

Mitigations

  • Per-link-type transitive closure: walk refines/supersedes/implements to fixpoint; related/contradicts zero hops unless tag-matched.
  • Relevance scoring with a token budget, not hop count. One tunable place.
  • Hybrid push-then-search: structured expand(atom_id, reason), not free-form pull.
  • Selection traces as first-class artifact per pod (considered / included / dropped, with reasons).
3.8 — HIGH PROBABILITY · DESIGN-FATAL

Too-late paradox: atoms launder agent guesses into requirements

Failure narrative

Month 1: first pods work greenfield areas, make reasonable-but-arbitrary choices (retry with exponential backoff, 5 attempts), and distill captures these as atoms: "constraint: payment retries use exponential backoff, max 5 attempts." A reviewer, lacking strong opinions on a decision they never made, approves. The atom is canonical.

Month 3: a second pod retrieves the atom, treats it as load-bearing intent, builds around it. The retry policy ossifies — not because anyone designed it, but because Claude picked it and nobody pushed back during distill review.

Month 6: an architect proposes changing retry behavior for a real business reason. They discover the codebase is shaped around an atom that encodes a decision no human ever made. The "intent corpus" is an archaeology of agent improvisation, retroactively legitimized by tired PR reviewers. The system has laundered guesses into requirements.

Falsified assumptions

  • Distill reviewers reject arbitrary agent choices — they can't distinguish "agent chose this" from "this was required" without redoing the analysis.
  • Pre-existing design docs seed the corpus organically — nobody backfills.
  • "Intent" is recoverable from outcomes — you can extract what was done, not why it had to be that way.
  • Greenfield is rare — in a growing codebase it's the majority of high-stakes work.
  • Human approval equals human authorship — under review fatigue, closer to rubber-stamping.

Early warning signs

  • Human-authored : distill-authored ratio trending to zero (target >30% human in active areas).
  • Atoms reading like postmortems ("we used X because it worked") instead of directives.
  • docs/decisions/ and atom corpus diverging; no cross-references.
  • New-subsystem pods producing 10+ atoms in first run — high improvisation, not high learning.
  • Atoms getting "corrected" by later atoms instead of revised.

Mitigations

  • Pre-work atom capture in /prep — interview produces atoms before the pod runs, not just an envelope. Distill refines, doesn't originate.
  • Provenance tagging: source: human-prep | distill-approved | doc-import. Retrieval can weight or filter by provenance.
  • Seed import from docs/decisions/ and ADRs so greenfield isn't empty.
  • Distill quarantine for first-pod-in-area: atoms enter a provisional tier; require a second pod or human design review to promote.

Pre-launch checklist

Single most important pre-launch action

Ship the ablation harness before atoms reach any real user. If injection doesn't move the success-rate needle on a benchmark you trust, no polish saves the design. Measurement before mechanism.

Before any user sees an atom

  • Ablation harness running, producing daily success-rate delta with vs. without atoms (n ≥ 50 frozen benchmark tasks).
  • Kill criteria written down with explicit numbers (suggested: retire if delta < +3pp at day 60, or < 0pp in any week after corpus > 30 atoms).
  • Exploration-ratio metric instrumented; alert threshold 1.5×.
  • Pilot repo selected — NOT autopod itself. Too close to home hides ergonomic problems.
  • Default OFF for simple task classes.

Schema and lint

  • Launch with TWO kinds: rationale and constraint. Defer intent.
  • Shape lint: reject >1 H2, >N top-level bullets, >1 kind, >40 lines.
  • claim: field required; LLM CI check validates body supports it.
  • Symbol-based anchors (tree-sitter) as primary; path-only deprecated for new atoms.
  • Closed-vocabulary tags and subsystems; taxonomy update is an explicit PR.
  • source (provenance) field required on every atom.

Workflow

  • Pre-work atom capture in /prep producing atoms before the pod runs.
  • Seed import from existing docs/decisions/ / ADRs / architecture docs.
  • Agent proposes atom edits in the same PR as code edits.
  • AST-symbol staleness detection via tree-sitter; surfaces in PR comments.
  • First-pod quarantine for atoms authored in virgin areas.
  • Git hook for anchor rewriting on rename.

Selector

  • Per-link-type transitive closure, not uniform one-hop.
  • Relevance scoring with token budget, not hop count.
  • Selection traces emitted per pod (atoms considered, included, dropped, with reasons).
  • Retrieval anti-incentive for long atoms (length penalty in scoring).

Telemetry SLOs (define before launch, monitor weekly)

  • Gap rate < 10% by day 60.
  • Noise rate < 30% by day 60.
  • Pull-tool invocation rate < 15% (push-primary discipline holding).
  • Median atom length p50 < 30 lines, p90 < 50 lines.
  • Orphan anchor rate < 5%.
  • Human-authored atom share > 30% in active subsystems.
  • Distill-proposed atom merge rate > 40%.
  • Exploration-ratio (atoms-on / atoms-off) < 1.3.
  • Success-rate delta with atoms > +3pp.

Decision gates

  • Week 2: telemetry pipeline producing all metrics. If not, halt content rollout.
  • Day 30: ablation delta ≥ 0pp. If negative, halt and diagnose.
  • Day 60: ablation delta ≥ +3pp AND noise/gap SLOs green. If not, redesign or retire.
  • Day 90: if all hold and ≥1 non-autopod consumer is producing atoms, begin standalone CLI extraction. Otherwise defer.

Closing

The premortem surfaces one design-fatal risk (empirical disproof of selective injection) and two slow-fade risks (curation collapse, too-late paradox) that together account for most likely-failure outcomes. The mitigations cluster into a small set of architectural moves: gate on ablation eval, default off for simple tasks, ship telemetry first, symbol-based anchors, provenance tagging, shape-lint not size-lint, two kinds not three, per-link-type transitive closure, and pre-work capture in /prep.

The single most important pre-launch action is the ablation eval. Ship the measurement before the mechanism.
